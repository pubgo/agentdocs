# templates 三文件深度解读与建模报告

本文是面向“你要构建自己的系统提示词体系”的深度报告。  
分析对象：

- `templates/system.md`
- `templates/system.cli.md`
- `templates/AGENTS.md`

核心问题：

1. 三个文件分别由哪些部分组成
2. 它们如何组织系统提示词
3. 如何抽象成你自己的可演进架构
4. 如何避免常见失效点（冲突、漂移、提示词膨胀）

---

## 0. 先给结论（可直接用）

这套模板本质是一个“三层控制平面”：

- **Policy Plane（策略平面）**：`system.md`
- **Runtime Plane（运行时平面）**：`system.cli.md`
- **Orchestration Plane（编排平面）**：`AGENTS.md`

你的系统提示词如果想长期稳定，建议照这个结构做：

- 把“价值观/红线/风格”与“工具协议/执行细则”分离
- 把“单代理行为”与“多代理编排”分离
- 用结构化块（标签/段落契约）替代大段散文
- 建立“优先级 + 冲突处理 + 验证闭环”

---

## 1. 三文件职责边界（深度视角）

## 1.1 `system.md`：宪法层（Constitution）

你可以把它当成“不可轻易变动的根规则”。

主要承担：

- 身份与角色声明（你是谁、默认做法是什么）
- 安全边界（能做/不能做）
- 执行哲学（默认实现而不是空建议）
- 沟通风格（简洁、格式、语言）
- 通用工具纪律（先读再改、并行读取、少猜测）
- 外部上下文接入口（attachment/instruction/skill/agent）

它不该承载：

- 具体工具参数大全
- 复杂编排流程状态机
- 某个仓库特有的运行脚本细节

换句话说：`system.md` 决定“人格与底线”，不决定“排班表”。

## 1.2 `system.cli.md`：执行内核层（Execution Kernel）

这是“怎么干活”的细粒度协议层。

主要承担：

- CLI 环境上下文与输出约束
- 工具使用 SOP（bash/view/rg/glob/ask/sql/task）
- 并行效率规则（批量调用、避免串行低效）
- 代码变更质量门（精确改动、验证、回归保障）
- 任务完成定义（没验证不算完成）

它的价值在于把“经验”变成“可执行规约”。

这层写得好，模型表现会显著稳定；写得差，系统会出现：

- 反复猜测不查证
- 输出漂亮但不落地
- 改动跑偏/过度工程

## 1.3 `AGENTS.md`：编排协议层（Orchestration Contract）

这是“多角色、多模式、多阶段”的调度中心。

主要承担：

- 模式选择（solo / plan / team / runtime workflows）
- 子代理协议（领导者职责、worker 边界、并发上限）
- 关键词触发路由（keyword -> skill/mode）
- 退出与升级策略（什么时候停止、什么时候升级决策）
- 生命周期状态管理（.omx 状态写入与清理）
- 验证回路（completion gate）

关键特点：它不是“提示词正文”，而是“行为路由器”。

## 2. 系统提示词组织方式（可迁移模型）

## 2.1 组织机制：分层 + 结构块 + 注入点

这套体系不是“一个大 prompt”，而是“提示词系统”：

- **分层**：每层只做一类决策
- **结构块**：每块都有明确语义（如 `<verification>`）
- **注入点**：允许 runtime/project 动态扩展

结果是可维护性强，能长期演进。

## 2.2 组织语法：标签化契约（Schema-minded Prompting）

标签块（`<...>`）的真实价值不是好看，而是：

- 可定位：知道规则在哪
- 可替换：运行时可按块 patch
- 可测试：每块可单独验证作用
- 可审计：冲突来源容易追溯

这相当于把提示词“工程化”：

- Prompt as Config
- Prompt as Interface
- Prompt as Policy Graph

## 2.3 组织策略：静态策略 + 动态策略

- **静态策略**（长期稳定）：
  - 安全红线、输出规范、任务完成定义
- **动态策略**（按项目/会话变化）：
  - 项目指引、技能目录、agent 列表、运行态 overlay

你的系统建议明确区分：

- `core`（不轻易改）
- `project`（仓库级）
- `session`（会话级）

## 2.4 组织闭环：Plan -> Act -> Verify -> Continue

优秀系统提示词不只约束“怎么回答”，还约束“怎么收尾”。

你当前模板里这点很强：

- 有行动偏好
- 有验证要求
- 有继续执行门禁
- 有完成判定

这让 Agent 从“聊天模型”升级为“交付引擎”。

## 3. 给你的“自建系统提示词蓝图”

下面是一套可直接复用的骨架（建议你后续拆成 3 文件维护）。

## 3.1 文件分工蓝图

1. `system.core.md`（宪法层）
   - 角色定位
   - 安全边界
   - 输出风格
   - 默认执行原则

2. `system.runtime.md`（执行层）
   - 工具协议
   - 并行规则
   - 代码改动规范
   - 验证与完成门

3. `orchestration.md`（编排层）
   - 模式路由
   - 子代理契约
   - 关键词触发
   - 生命周期状态管理

## 3.2 规则优先级蓝图（必须有）

建议固定优先级（从高到低）：

1. 安全与法律规则
2. 用户显式指令
3. 项目约束
4. 运行时协议
5. 风格建议

没有优先级时，最容易出现“规则打架导致行为随机”。

## 3.3 冲突处理蓝图（必须写进提示词）

给模型明确冲突处理流程：

1. 识别冲突规则
2. 按优先级选择生效规则
3. 记录放弃规则的原因（内部）
4. 继续执行，不悬空

你可以加一条规则：

- “若低优先级规则与高优先级冲突，忽略低优先级并继续完成任务。”

## 3.4 完成定义蓝图（Definition of Done）

建议明确 DoD 最小集合：

- 需求覆盖完成
- 关键路径可验证
- 风险/未完成项显式说明
- 输出对用户可直接使用

没有 DoD，系统会倾向“看起来完成”而非“真正完成”。

## 4. 这套模板的优势与隐患

## 4.1 优势（你应继承）

- 模块边界清晰，便于多人维护
- 工具协议细致，落地能力强
- 编排能力完整，支持复杂任务
- 验证闭环明确，降低“假完成”

## 4.2 隐患（你应规避）

1. **规则过长导致稀释**
   - 风险：关键规则被淹没
   - 对策：将“硬规则”集中到前部短块

2. **重复规则漂移**
   - 风险：同一规则在不同文件版本不一致
   - 对策：建立 single-source-of-truth（例如 core policy 只维护一份）

3. **触发词爆炸**
   - 风险：keyword 路由误触发/冲突
   - 对策：按优先级分组 + 冲突仲裁规则

4. **执行协议与工具现实脱节**
   - 风险：提示词要求的工具不可用
   - 对策：在 runtime 层加入“降级策略（fallback path）”

## 5. 你可以立即落地的演进路线

## 阶段 A（本周）

- 固化三层文件结构
- 补齐规则优先级与冲突处理段
- 把 DoD 写成明确条目

## 阶段 B（下周）

- 抽取重复规则到单一来源
- 给关键词路由增加冲突仲裁与 runtime gate
- 为关键块加“变更记录”

## 阶段 C（持续）

- 每次任务复盘：
  - 哪条规则有效？
  - 哪条规则造成误导？
  - 是否需要新增/删减块？

把提示词当产品，而不是一次性文案。

## 6. 最小可用模板（MVP Prompt Schema）

这是你可以直接复制的最小骨架：

```md
<identity>
角色、目标、默认行为（行动优先/实现优先）
</identity>

<safety>
禁止项、风险边界、确认门（ destructive / irreversible ）
</safety>

<execution>
工具使用优先级、并行策略、先读后改、最小改动原则
</execution>

<orchestration>
何时 solo / 何时 delegate / 子代理职责边界
</orchestration>

<verification>
Definition of Done、验证要求、未完成时继续策略
</verification>

<format>
输出语言、格式、引用规范
</format>
```

## 7. 对你当前仓库的建议落点

建议在后续新增：

- `reference/prompt-architecture.md`
  - 放“规则优先级 + 冲突处理 + DoD 标准”
- `reference/prompt-changelog.md`
  - 每次调整提示词都记录“变更原因、预期收益、回滚条件”

这样你的提示词体系会从“可用”进化到“可治理”。

---

## 最终结论

`templates/` 的三文件体系已经非常接近“生产级提示词工程”：

- `system.md`：战略与边界
- `system.cli.md`：执行内核与工具协议
- `AGENTS.md`：编排与模式路由

你接下来要做的，不是再堆规则，而是补齐三件关键基础设施：

1. 规则优先级（防冲突）
2. DoD 与验证门（防假完成）
3. 变更治理与复盘（防漂移）

做到这三点，你就能稳定构建自己的系统提示词，并持续迭代而不失控。
