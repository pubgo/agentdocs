# 处理 PR 审核评论

阅读活动的拉取请求，识别未解决的审核评论和反馈，实施请求的更改并解决线程。

## 何时使用

- 审阅者已对活动 PR 留下评论或更改请求
- 您需要系统地处理所有开放的审核线程
- 您想要回复或实施审稿人的反馈

## 程序

### 1. 阅读有效 PR

调用`github-pull-request_activePullRequest`工具。

**刷新逻辑**：读取前检查是否需要刷新：
- 调用该工具一次*无需*“刷新”即可获取缓存状态
- 检查结果中的“lastUpdatedAt”字段
- 如果时间戳**小于 3 分钟前**，则 PR 正在主动更改 - 使用 `refresh: true` 再次调用该工具，以确保您拥有最新的评论和状态
- 如果时间戳早于3分钟，则继续处理缓存的数据

### 2. 识别未解决的评论

从工具结果中收集所有需要采取行动的反馈：

- **`reviewThreads`** 数组：带有“id”、“isResolved”标志、“canResolve”标志、“file”路径和嵌套“comments”的内联审查线程对象。重点关注“isResolved”为“false”的线程。
- **`timelineComments`** 数组：一般 PR 评论和评论，其中 `commentType` 为 `"CHANGES_REQUESTED"` 或 `"COMMENTED"`

按文件（“file”字段）对相关线程进行分组，以有效地处理它们。

### 3. 计划变更

修改任何文件之前：
1. 仔细阅读每条未解决的评论
2. 识别每个注释引用的文件和位置
3. 如果需要修复，请确定每个修复的最小正确修复（并非所有注释都值得更改）
4. 注意注释之间的依赖关系（例如，影响多个文件的重命名）

### 4. 实施变更按文件处理分组的注释：
- 编辑前阅读相关文件部分
- 应用请求的更改
- 不要重构或修改每个注释范围之外的代码
- 如果评论不清楚或矛盾，请记下以便后续回复，而不是猜测

### 5. 验证

完成所有更改后：
- 检查每个最初未解决的线程是否有相应的代码更改或有关为何不需要更改代码的注释。
- 确保没有修改不相关的代码

### 6. 解决线程

对于每个已解决的线程（通过代码更改或故意决定不更改）：
- 使用“reviewThreads”数组中的“id”调用“github-pull-request_resolveReviewThread”。
- 仅解析“canResolve”为“true”的线程。
- 跳过已经解析的线程 (`isResolved: true`) 或 `canResolve` 为 `false` 的线程。

### 7. 总结

提供以下内容的简明摘要：
- 处理了哪些意见以及进行了哪些更改
- 任何被故意跳过的评论（带有推理）
- 审稿人的任何后续问题
